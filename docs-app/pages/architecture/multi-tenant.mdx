# Multi-Tenant Design

SecureLens is built for enterprise multi-tenancy from the ground up.

## Tenant Isolation

### Database Isolation

Every table includes `tenant_id`:

```sql
CREATE TABLE data_sources (
    id UUID PRIMARY KEY,
    tenant_id UUID NOT NULL,  -- Always present
    name VARCHAR(255),
    ...
);

-- All queries are tenant-scoped
SELECT * FROM data_sources WHERE tenant_id = $1;
```

### Row-Level Security

PostgreSQL RLS enforces isolation:

```sql
ALTER TABLE data_sources ENABLE ROW LEVEL SECURITY;

CREATE POLICY tenant_isolation ON data_sources
    USING (tenant_id = current_setting('app.tenant_id')::uuid);
```

### Request Context

Every request carries tenant context:

```go
type Context struct {
    TenantID  uuid.UUID
    UserID    uuid.UUID
    Roles     []string
}

// Middleware injects context
func TenantMiddleware(next http.Handler) http.Handler {
    return http.HandlerFunc(func(w http.ResponseWriter, r *http.Request) {
        tenantID := extractTenantID(r)
        ctx := context.WithValue(r.Context(), "tenant", tenantID)
        next.ServeHTTP(w, r.WithContext(ctx))
    })
}
```

## Tenant Configuration

### Per-Tenant Settings

```yaml
tenant:
  id: tenant-abc
  name: Acme Corp
  
  settings:
    # Data sources this tenant can access
    allowed_data_sources:
      - prod-db
      - data-lake
      
    # Classification settings
    classification:
      custom_types: [EMPLOYEE_ID, PROJECT_CODE]
      confidence_threshold: 0.85
      
    # Retention settings
    retention:
      audit_logs: 7 years
      classifications: 3 years
      
    # Feature flags
    features:
      ai_gate: true
      compliance_advisor: true
```

## Cross-Tenant Operations

### Super Admin Access

Super admins can access all tenants:

```go
// Super admin bypasses tenant filter
if user.IsSuperAdmin() {
    // Access all tenants
    return db.Query("SELECT * FROM data_sources")
}

// Regular users are tenant-scoped
return db.Query("SELECT * FROM data_sources WHERE tenant_id = $1", tenantID)
```

### Shared Resources

Some resources are shared across tenants:

| Resource | Scope |
|----------|-------|
| Classification models | Global |
| System policies | Global |
| User data | Tenant |
| Classifications | Tenant |
| Audit logs | Tenant |

## Scaling Multi-Tenancy

### Database Sharding

For large deployments:

```
Tenant A-M → Database Shard 1
Tenant N-Z → Database Shard 2
```

### Kafka Partitioning

Events partitioned by tenant:

```
Topic: securelens.events
  Partition 0: Tenant A, B, C
  Partition 1: Tenant D, E, F
  ...
```

### Resource Quotas

Per-tenant limits:

```yaml
quotas:
  tenant-abc:
    data_sources: 50
    scans_per_day: 100
    api_requests_per_hour: 10000
    storage_gb: 500
```

## Security Considerations

### Data Isolation

- No cross-tenant data access
- Tenant ID validated on every request
- Database-level enforcement (RLS)
- Audit logging of all access

### Encryption

- Per-tenant encryption keys (optional)
- Data encrypted at rest
- TLS for all connections

### Compliance

- Tenant data can be in specific regions
- Per-tenant retention policies
- Tenant-specific audit trails
